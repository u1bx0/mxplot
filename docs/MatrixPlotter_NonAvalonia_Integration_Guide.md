# Hosting MatrixPlotter in WinForms / WPF

**Created**: 2026-04-24  
**Target version**: `MxPlot.UI.Avalonia` (Avalonia 11.3.14), .NET 10

---

> **📌 Scope of this guide**  
> This guide primarily covers calling `MatrixPlotter` (an Avalonia-based **standalone window**) from WinForms or WPF.  
> Embedding `MxView` directly inside a WinForms panel or WPF window is also possible using  
> `Avalonia.Win32.Interoperability` (`WinFormsAvaloniaControlHost`),  
> which is **available from Avalonia 11.3.x** (stable).  
> See [Section 9](#9-direct-mxview-embedding-winforms--wpf) for details.

---

## Overview

`MxPlot.UI.Avalonia.Views.MatrixPlotter` is an Avalonia-based window.  
Both WinForms and WPF use a Win32 message pump, so they can coexist with Avalonia.  
Using `SetupWithoutStarting()` prevents Avalonia from starting its own loop and shares the host pump instead.  
Both frameworks run on the same UI thread, so there is no conflict between `Dispatcher` and `SynchronizationContext`.

| | WinForms | WPF (OnStartup) | WPF (Generic Host) |
|---|---|---|---|
| Initialization point | Top of `Program.Main()` | Top of `App.OnStartup()` | Top of `Program.Main()` |
| Entry point | Explicit `Main()` | Auto-generated by `App.xaml` | `Program.cs` created manually |
| Window opening style | Mainly code-behind | Code-behind / MVVM | MVVM (DI-based) |

---

## 1. Project Reference Setup

Add the following to your WinForms / WPF project `.csproj`.

```xml
<ItemGroup>
  <ProjectReference Include="..\MxPlot.UI.Avalonia\MxPlot.UI.Avalonia.csproj" />
</ItemGroup>

<ItemGroup>
  <!-- Avalonia Windows backend (Win32 + Skia) -->
  <!-- ⚠️ Version must exactly match the Avalonia dependency of MxPlot.UI.Avalonia -->
  <PackageReference Include="Avalonia.Win32" Version="11.3.14" />
  <PackageReference Include="Avalonia.Skia" Version="11.3.14" />
</ItemGroup>
```

> **⚠️ Version mismatch causes a crash before startup**  
> If `Avalonia.Win32` / `Avalonia.Skia` differ from the Avalonia version used by `MxPlot.UI.Avalonia`,  
> the process terminates with exit code `0xe0434352` **before** reaching `Main()` / `OnStartup()`.  
> This error cannot be caught with try-catch.

> **Do not use `Avalonia.Desktop`**  
> It is a meta-package bundling Win32/macOS/Linux, and it transitively introduces a  
> vulnerability in `Tmds.DBus.Protocol` (GHSA-xrw6-gwf8-vvr9, High).  
> `Avalonia.Win32` + `Avalonia.Skia` alone are sufficient for Windows.

## 2. Application Class Definition (Not Required)

`MxPlot.UI.Avalonia` exposes `MxPlotHostApplication` (a subclass of `Avalonia.Application`).  
You do **not** need to define your own `Application` subclass in the host project.

> **Projects that already have an Avalonia Application** (e.g., referencing `MxPlot.App`)  
> should keep their existing `AppBuilder` initialization rather than using `MxPlotHostApplication`.

## 3. Initializing Avalonia

### WinForms (Program.cs)

```csharp
using Avalonia;
using MxPlot.UI.Avalonia;

[STAThread]
static void Main()
{
    // ① Integrate Avalonia into the WinForms message loop
    AppBuilder.Configure<MxPlotHostApplication>()
        .UseWin32()
        .UseSkia()
        .SetupWithoutStarting();

    // ② Normal WinForms initialization
    Application.EnableVisualStyles();
    Application.SetCompatibleTextRenderingDefault(false);
    Application.SetHighDpiMode(HighDpiMode.SystemAware);
    ApplicationConfiguration.Initialize();
    Application.Run(new MainForm());
}
```

### WPF (App.xaml.cs) — OnStartup pattern (default configuration)

The Visual Studio WPF template auto-generates the entry point, so there is no `Program.cs`.  
Write the Avalonia initialization in `OnStartup` inside `App.xaml.cs`.

> **Watch out for `StartupUri` in `App.xaml`**  
> The template default sets `StartupUri="MainWindow.xaml"`.  
> If you create and `Show()` `MainWindow` manually, remove `StartupUri` to avoid **two windows opening**.

```csharp
// App.xaml.cs
using Avalonia;
using MxPlot.UI.Avalonia;

public partial class App : Application
{
    protected override void OnStartup(StartupEventArgs e)
    {
        // ① Integrate Avalonia into the WPF message loop (call before base)
        AppBuilder.Configure<MxPlotHostApplication>()
            .UseWin32()
            .UseSkia()
            .SetupWithoutStarting();

        base.OnStartup(e);
        // If using StartupUri, finish here.
        // Set DataContext in the MainWindow constructor or code-behind.
    }
}
```

---

### WPF (Program.cs) — Generic Host pattern (medium to large scale)

Professional projects increasingly use `Microsoft.Extensions.Hosting` (Generic Host) with a manually-defined `Program.cs`.  
It integrates logging, configuration, and DI using the same pattern as ASP.NET Core.

**Preparation: enabling `Program.cs`**

WPF's `App.g.cs` auto-generates `Main()`, which conflicts with a manual one.  
Add `<StartupObject>` to `.csproj` to disable the auto-generated `Main()`:

```xml
<!-- .csproj -->
<PropertyGroup>
  <StartupObject>YourNamespace.Program</StartupObject>
</PropertyGroup>
```

Also remove `StartupUri` from `App.xaml`:

```xml
<!-- App.xaml — remove StartupUri -->
<Application x:Class="YourNamespace.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
  <Application.Resources />
</Application>
```

**Program.cs**

```csharp
using Avalonia;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using MxPlot.UI.Avalonia;

namespace YourNamespace;

public class Program
{
    [STAThread]
    public static void Main(string[] args)
    {
        // ① Integrate Avalonia into the WPF message loop
        AppBuilder.Configure<MxPlotHostApplication>()
            .UseWin32()
            .UseSkia()
            .SetupWithoutStarting();

        // ② Register services with Generic Host
        var host = Host.CreateDefaultBuilder(args)
            .ConfigureServices(services =>
            {
                services.AddSingleton<IMatrixPlotterService, MatrixPlotterService>();
                services.AddSingleton<MainViewModel>();
                services.AddSingleton<MainWindow>();
            })
            .Build();

        // ③ Start the WPF application
        var app = new App();
        app.InitializeComponent();
        var mainWindow = host.Services.GetRequiredService<MainWindow>();
        mainWindow.DataContext = host.Services.GetRequiredService<MainViewModel>();
        app.Run(mainWindow);
    }
}
```

> **Which pattern to choose**
>
> | | OnStartup pattern | Generic Host pattern |
> |---|---|---|
> | `Program.cs` | Not needed (default) | Must be created manually |
> | Additional packages | None | `Microsoft.Extensions.Hosting` |
> | Logging / configuration | Arrange separately | Built in automatically |
> | Recommended for | Small to medium tools | Medium to large / team development |

## 4. Opening MatrixPlotter

### Code-behind (WinForms / WPF common)

```csharp
using MxPlot.Core;
using MxPlot.Core.Imaging;
using MxPlot.UI.Avalonia.Views;

// ── Minimal setup ─────────────────────────────────────────────
var data = MatrixData<float>.Create(512, 512);
// ... write data into data ...

MatrixPlotter.Create(data).Show();

// ── With LUT and title ────────────────────────────────────────
MatrixPlotter.Create(data, ColorThemes.Jet, "Measurement Result #1").Show();

// ── Hold a reference for later operations ────────────────────
var plotter = MatrixPlotter.Create(data, ColorThemes.Grayscale, "Real-time View");
plotter.Show();
```

Opening from a background thread:

```csharp
await Task.Run(async () =>
{
    ProcessData(data);
    await Avalonia.Threading.Dispatcher.UIThread.InvokeAsync(()
        => MatrixPlotter.Create(data).Show());
});
```

### MVVM (WPF)

#### Pattern ①: No DI — CommunityToolkit.Mvvm only (minimal)

`MatrixPlotter.Create()` is a static method, so it can be called directly without service abstraction or DI.

```csharp
using Avalonia.Threading;
using CommunityToolkit.Mvvm.ComponentModel;
using CommunityToolkit.Mvvm.Input;
using MxPlot.UI.Avalonia.Views;

public partial class MainViewModel : ObservableObject
{
    [RelayCommand]
    private async Task OpenPlotterAsync()
    {
        // Run heavy processing on a background thread
        var data = await Task.Run(() => LoadData());

        // Create and show MatrixPlotter on the Avalonia UI thread
        await Dispatcher.UIThread.InvokeAsync(()
            => MatrixPlotter.Create(data, "Analysis Result").Show());
    }
}
```

Set `StartupUri` in `App.xaml` and pass the ViewModel from code-behind:

```csharp
// App.xaml.cs
protected override void OnStartup(StartupEventArgs e)
{
    AppBuilder.Configure<MxPlotHostApplication>()
        .UseWin32()
        .UseSkia()
        .SetupWithoutStarting();

    base.OnStartup(e);
    // When App.xaml StartupUri="MainWindow.xaml" auto-creates the window,
    // set DataContext in the MainWindow constructor or code-behind.
}
```

```csharp
// MainWindow.xaml.cs
public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();
        DataContext = new MainViewModel();
    }
}
```

#### Pattern ②: Service interface + DI (testability-first)

To avoid calling `Dispatcher.UIThread` directly from the ViewModel and enable unit testing:

**Service interface**

```csharp
public interface IMatrixPlotterService
{
    void Show(IMatrixData data, string title = "");
    Task ShowAsync(IMatrixData data, string title = "");
}
```

**Implementation**

```csharp
using Avalonia.Threading;
using MxPlot.UI.Avalonia.Views;

public class MatrixPlotterService : IMatrixPlotterService
{
    public void Show(IMatrixData data, string title = "")
    {
        Dispatcher.UIThread.VerifyAccess();
        MatrixPlotter.Create(data, title: title).Show();
    }

    public Task ShowAsync(IMatrixData data, string title = "")
        => Dispatcher.UIThread.InvokeAsync(()
               => MatrixPlotter.Create(data, title: title).Show()
           ).GetTask();
}
```

**ViewModel**

```csharp
public partial class MainViewModel : ObservableObject
{
    private readonly IMatrixPlotterService _plotterService;

    public MainViewModel(IMatrixPlotterService plotterService)
    {
        _plotterService = plotterService;
    }

    [RelayCommand]
    private async Task OpenPlotterAsync()
    {
        var data = await Task.Run(() => LoadData());
        await _plotterService.ShowAsync(data, "Analysis Result");
    }
}
```

**DI registration (App.xaml.cs)**

```csharp
protected override void OnStartup(StartupEventArgs e)
{
    AppBuilder.Configure<MxPlotHostApplication>()
        .UseWin32()
        .UseSkia()
        .SetupWithoutStarting();

    var services = new ServiceCollection();
    services.AddSingleton<IMatrixPlotterService, MatrixPlotterService>();
    services.AddTransient<MainViewModel>();

    var provider = services.BuildServiceProvider();
    new MainWindow { DataContext = provider.GetRequiredService<MainViewModel>() }.Show();

    base.OnStartup(e);
}
```

> **Which pattern to choose**
>
> | | Pattern ① | Pattern ② |
> |---|---|---|
> | Additional packages | CommunityToolkit.Mvvm only | + Microsoft.Extensions.DependencyInjection |
> | ViewModel testability | Requires `Dispatcher.UIThread` initialization | Can be replaced with a mock |
> | Recommended for | Small / one-off tools | Medium+ projects with unit tests |

## 5. Updating Data and Refreshing

After modifying data, call `plotter.Refresh()` to trigger a redraw.

### Pattern A: Indexer (most common)

```csharp
data[ix, iy] = newValue;   // indexer automatically invalidates the internal cache
plotter.Refresh();          // safe to call from either UI or non-UI thread
```

### Pattern B: Direct array write via GetArray

```csharp
var arr = data.GetArray();          // cache is automatically invalidated at this point
arr[iy * width + ix] = newValue;
plotter.Refresh();
```

### Pattern C: GetArray → GetValueRange → external modification (edge case)

```csharp
var arr = data.GetArray();
var range = data.GetValueRange();   // cache confirmed (IsValid = true)
arr[iy * width + ix] = newValue;    // → cache becomes stale

data.Invalidate();                  // explicit Invalidate required only in this case
plotter.Refresh();
```

---

## 6. Thread-Safe Usage

`plotter.Refresh()` is **safe to call from a non-UI thread**.  
Internally it checks the thread via `Dispatcher.UIThread.CheckAccess()` and schedules via `Post()` when called off the UI thread.

```csharp
await Task.Run(() =>
{
    ProcessData(data);
    plotter.Refresh();   // fire-and-forget, queued to UI thread
});
```

> If you need to wait for rendering to complete, use `Dispatcher.UIThread.InvokeAsync()` directly.

---

## 7. High-Frequency Update Pattern (Timer-Based)

Calling `Refresh()` on every update can flood the UI thread queue.  
The best practice is to use a **dirty flag + timer** to throttle the update rate.

```csharp
private volatile bool _isDirty = false;

// Form initialization (WinForms example; WPF uses DispatcherTimer similarly)
void InitRefreshTimer()
{
    var timer = new System.Windows.Forms.Timer { Interval = 33 }; // ~30 fps
    timer.Tick += (_, _) =>
    {
        if (_isDirty) { _isDirty = false; plotter.Refresh(); }
    };
    timer.Start();
}

void OnNewSample(int ix, int iy, float value)
{
    data[ix, iy] = value;
    _isDirty = true;   // do not call Refresh here
}
```

## 8. Notes

| Item | Detail |
|---|---|
| Initialization order | `SetupWithoutStarting()` must be called **before** the host app's initialization (`Application.Run` / `base.OnStartup`) |
| Threading | `MatrixPlotter.Create().Show()` must be called from the UI thread. Only `Refresh()` is thread-safe |
| WPF Dispatcher | WPF's `Application.Current.Dispatcher` and Avalonia's `Dispatcher.UIThread` refer to the same thread |
| `data.Invalidate()` | Normally not required. The indexer and `GetArray()` automatically invalidate the internal cache |
| DPI (WinForms) | Always set `Application.SetHighDpiMode(HighDpiMode.SystemAware)` |
| Version pinning | `Avalonia.Win32` / `Avalonia.Skia` must match the Avalonia version used by `MxPlot.UI.Avalonia` |
| `Avalonia.Desktop` | **Do not use** (introduces `Tmds.DBus.Protocol` vulnerability GHSA-xrw6-gwf8-vvr9) |

## 9. Direct MxView Embedding (WinForms / WPF)

Instead of opening `MatrixPlotter` as a standalone window, you can embed `MxView` directly
inside a WinForms `Panel` or a WPF window using `WinFormsAvaloniaControlHost` from
`Avalonia.Win32.Interoperability`. This package is available as a **stable release from
Avalonia 11.3.x**.

### Additional package reference

Add the following to your `.csproj`:

```xml
<ItemGroup>
  <PackageReference Include="Avalonia.Win32" Version="11.3.14" />
  <PackageReference Include="Avalonia.Skia" Version="11.3.14" />
  <PackageReference Include="Avalonia.Win32.Interoperability" Version="11.3.14" />
</ItemGroup>
```

> Version must exactly match the Avalonia version used by `MxPlot.UI.Avalonia`.

---

### 9-A. WinForms

```csharp
// Form1.cs
using Avalonia;
using Avalonia.Win32.Interoperability;
using MxPlot.Core;
using MxPlot.Core.Imaging;
using MxPlot.UI.Avalonia;
using MxPlot.UI.Avalonia.Controls;

public partial class Form1 : Form
{
    private WinFormsAvaloniaControlHost _avaloniaHost = null!;

    protected override void OnLoad(EventArgs e)
    {
        base.OnLoad(e);

        AppBuilder.Configure<MxPlotHostApplication>()
            .UseWin32()
            .UseSkia()
            .SetupWithoutStarting();

        var data = new MatrixData<float>(256, 256);
        data.Set((ix, iy, x, y) => (float)(Math.Sin(x * 0.05) * Math.Cos(y * 0.05)));

        var mxView = new MxView
        {
            MatrixData = data,
            Lut = ColorThemes.Get("Jet"),
        };

        _avaloniaHost = new WinFormsAvaloniaControlHost
        {
            Content = mxView,
            Dock = DockStyle.Fill,
        };
        panel1.Controls.Add(_avaloniaHost);
    }

    protected override void Dispose(bool disposing)
    {
        if (disposing)
            _avaloniaHost?.Dispose();
        if (disposing && components != null)
            components.Dispose();
        base.Dispose(disposing);
    }
}
```

### 9-B. WPF

Avalonia 11.3.x does not provide `WpfAvaloniaHost`. The only approach is a **3-layer stack**:  
`WPF Window` → `WindowsFormsHost` → `WinFormsAvaloniaControlHost` → `MxView`

**`.csproj` — enable both WPF and WinForms**

```xml
<PropertyGroup>
  <UseWPF>true</UseWPF>
  <UseWindowsForms>true</UseWindowsForms>  <!-- required for WindowsFormsHost -->
</PropertyGroup>
```

**`App.xaml.cs` — namespace collision workaround**

```csharp
using Avalonia;
using MxPlot.UI.Avalonia;

namespace MxPlotWPFEmbedTest;

public partial class App : System.Windows.Application
{
    protected override void OnStartup(System.Windows.StartupEventArgs e)
    {
        // Initialize Avalonia once, before any Avalonia control is created.
        AppBuilder.Configure<MxPlotHostApplication>()
            .UseWin32()
            .UseSkia()
            .SetupWithoutStarting();

        base.OnStartup(e);
    }
}
```

> **`using System.Windows;` causes CS0104**  
> `Avalonia.Application` and `System.Windows.Application` collide.  
> Omit `using System.Windows;` and use fully-qualified names instead
> (`System.Windows.Application`, `System.Windows.StartupEventArgs`).

**`MainWindow.xaml` — declare `WindowsFormsHost` in XAML**

```xml
<Window x:Class="MxPlotWPFEmbedTest.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:wfi="clr-namespace:System.Windows.Forms.Integration;assembly=WindowsFormsIntegration"
        Title="WPF Embed Test" Height="450" Width="800">
  <Grid>
    <wfi:WindowsFormsHost x:Name="wfHost" />
  </Grid>
</Window>
```

**`MainWindow.xaml.cs`**

```csharp
using Avalonia.Win32.Interoperability;
using MxPlot.Core;
using MxPlot.Core.Imaging;
using MxPlot.UI.Avalonia.Controls;
using System.Windows;

namespace MxPlotWPFEmbedTest;

public partial class MainWindow : Window
{
    private WinFormsAvaloniaControlHost _avaloniaHost = null!;

    public MainWindow()
    {
        InitializeComponent();

        var data = new MatrixData<float>(256, 256);
        data.Set((ix, iy, x, y) => (float)(Math.Sin(x * 0.05) * Math.Cos(y * 0.05)));

        var mxView = new MxView
        {
            MatrixData = data,
            Lut = ColorThemes.Get("Jet"),
        };

        _avaloniaHost = new WinFormsAvaloniaControlHost
        {
            Content = mxView,
            Dock = System.Windows.Forms.DockStyle.Fill,
        };
        wfHost.Child = _avaloniaHost;
    }
}
```

### Known limitation: ContextMenu items have no effect

`MxView` displays a right-click context menu (Crop, Export, etc.), but in direct embedding mode
the handlers that `MatrixPlotter` normally provides (e.g., `CropRequested`) are not wired up.
Menu items appear but do nothing. This is a known limitation of the current version.

| Scenario | ContextMenu behavior |
|---|---|
| `MatrixPlotter` standalone window | Fully functional (Crop, Export, etc.) |
| Direct embedding via `WinFormsAvaloniaControlHost` | Menu visible, handlers not connected — **no effect** |

## Appendix: ProfilePlotter — Displaying (X, Y) Series

`ProfilePlotter` is a line/scatter plotter that, like `MatrixPlotter`, operates as a standalone window.  
Once Avalonia is initialized (Section 3), it can be used the same way as `MatrixPlotter`.

### Main Classes

| Class | Location | Role |
|---|---|---|
| `ProfilePlotter` | `MxPlot.UI.Avalonia.Views` | Window host. Includes menu and info panel |
| `ProfilePlotControl` | `MxPlot.UI.Avalonia.Controls` | Plot rendering control (accessed via `ProfilePlotter.Plot`) |
| `PlotSeries` | `MxPlot.UI.Avalonia.Controls` | Single series data (point list + name + style) |
| `PlotStyle` | `MxPlot.UI.Avalonia.Controls` | `Line` / `Marker` / `MarkedLine` |

---

### Basic Usage

#### Single series

```csharp
using MxPlot.UI.Avalonia.Views;
using MxPlot.UI.Avalonia.Controls;

// Pass a list of (double X, double Y) tuples
var points = Enumerable.Range(0, 100)
    .Select(i => ((double)i, Math.Sin(i * 0.1)))
    .ToList();

new ProfilePlotter(points,
    name       : "sin(x)",
    xAxisLabel : "Index",
    yAxisLabel : "Value",
    title      : "Sine Wave").Show();
```

#### Multiple series (PlotSeries list)

```csharp
var series = new List<PlotSeries>
{
    new(points1, "sin",  PlotStyle.Line,       Colors.SteelBlue),
    new(points2, "cos",  PlotStyle.Line,       Colors.OrangeRed),
    new(points3, "data", PlotStyle.MarkedLine, lineWidth: 1.0),
};

new ProfilePlotter(series,
    xAxisLabel : "Time [s]",
    yAxisLabel : "Amplitude",
    title      : "Waveform Comparison").Show();
```

#### Multiple series (parallel point-list overload)

```csharp
var pointSets = new List<IReadOnlyList<(double, double)>> { points1, points2 };
var names     = new List<string> { "Channel A", "Channel B" };

new ProfilePlotter(pointSets, names,
    xAxisLabel : "Freq [Hz]",
    yAxisLabel : "Power",
    style      : PlotStyle.Line,
    title      : "Spectrum").Show();
```

---

### Data Updates (Real-time)

Hold a reference to `ProfilePlotter` and update it via `Plot.UpdatePoints()` or `Plot.UpdatePointsAndFit()`.

```csharp
var plotter = new ProfilePlotter(initialPoints, name: "Measurement");
plotter.Show();

// ── Update points only, keeping the current view range (fastest) ──────────
plotter.Plot.UpdatePoints(seriesIndex: 0, newPoints);

// ── Update points and fit the view to the data ────────────────────────────
plotter.Plot.UpdatePointsAndFit(seriesIndex: 0, newPoints);

// ── Replace all series ────────────────────────────────────────────────────
plotter.Plot.SetData(newSeriesList, "X", "Y", "New Title");
```

> **Threading**: `UpdatePoints` / `SetData` must be called from the Avalonia UI thread.  
> When calling from a background thread, wrap with `Dispatcher.UIThread.InvokeAsync`.

---

### Fixing Axis Ranges

```csharp
// Fix X axis to 0–100, Y axis automatic
plotter.Plot.XAxisFixed = true;
plotter.Plot.XFixedMin  = 0;
plotter.Plot.XFixedMax  = 100;

// Set known data range at once (skips O(N) data scan, fast)
plotter.Plot.SetViewRange(xRange: (0, 100), yRange: (-1.5, 1.5));
```

### PlotSeries Properties

| Property | Type | Default | Description |
|---|---|---|---|
| `Points` | `IReadOnlyList<(double X, double Y)>` | — | Point data |
| `Name` | `string` | `""` | Series name shown in the legend |
| `Style` | `PlotStyle` | `Line` | `Line` / `Marker` / `MarkedLine` |
| `Color` | `Color?` | `null` (auto palette) | Series color |
| `LineWidth` | `double` | `1.5` | Line width (dip) |
